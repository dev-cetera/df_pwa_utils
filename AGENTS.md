## Package

`df_pwa_utils` — a published Flutter package (pub.dev) providing PWA/web utilities that no-op on non-web platforms (including WASM). Public surface is intentionally tiny: `setToUrlPathStrategy()`, the `platformNavigator` singleton, and `normalizePathQuery()`.

## Common commands

```bash
dart analyze
flutter pub get
flutter pub publish --dry-run   # validate before release
```

There is no `test/` directory; this package currently ships without unit tests.

## Architecture

### Conditional platform implementations

Every platform-sensitive feature follows the same three-file pattern under `lib/src/<feature>/`:

- `any.dart` — public entrypoint, contains only: `export '_web.dart' if (dart.library.io) '_non_web.dart';`
- `_web.dart` — real implementation using `package:web` and `flutter_web_plugins`
- `_non_web.dart` — no-op stubs with identical API surface

This means consumers always import the same symbol; the compiler swaps the impl based on `dart.library.io`. When adding a new platform-specific API, follow this exact split — don't put `kIsWeb` branches in `any.dart`.

Note that `_web.dart` files re-export `_non_web.dart` so shared types (e.g. `PathUrlStrategy` stub) remain available where needed.

### `lib/src/_src.g.dart` is generated

This barrel file is auto-generated by [`df_generate_dart_indexes`](https://github.com/dev-cetera/df_generate_dart_indexes). Do not edit by hand. When adding new files under `lib/src/`, regenerate rather than appending exports manually.

### Platform navigator base contract

`PlatformNavigatorBase` (in `lib/src/platform_navigator/_platform_navigator_base.dart`) defines the abstract API. Both `WebNavigator` and the non-web `PlatformNavigator` implement it as singletons. The web impl manages a single shared `onPopState` subscription that is lazily started on first `addStateCallback` and cancelled when the last callback is removed — preserve this lifecycle when modifying.

The web `pushState`/`replaceState` and the pop-state handler all route paths through `urlStrategy?.prepareExternalUrl(...)` so behavior stays consistent with whichever `UrlStrategy` Flutter has installed. The pop-state handler also calls `replaceState` to canonicalize the URL after stripping `<base href>`.

## Release / CI workflow

Releases are driven by commit message prefixes on `main` (see `.github/workflows/process-package.yml`):

- `+<msg>` — CI runs `dart format`, `dart fix --apply`, regenerates README/CHANGELOG, bumps to the `pubspec.yaml` version, commits `ci: bump version to vX.Y.Z`, and pushes a `vX.Y.Z` tag.
- `++<msg>` — same as above and (when the release job is uncommented) creates a GitHub Release.
- Any other prefix — no release actions.

So when preparing a release, the typical flow is: bump `version:` in `pubspec.yaml` → commit with a leading `+` (or `++`). README is auto-regenerated from `_README_CONTENT.md` plus an upstream template; edit `_README_CONTENT.md`, not `README.md` directly (CI will overwrite it).

## Lint configuration
